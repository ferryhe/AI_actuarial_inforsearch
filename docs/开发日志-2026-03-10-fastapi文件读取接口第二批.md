# 2026-03-10 FastAPI 文件读取接口第二批开发日志

## 本轮目标

将仍由 React 直接依赖、但仍停留在 legacy Flask 的三条结构化读接口迁为 FastAPI 原生实现：

- `GET /api/sources`
- `GET /api/files/detail`
- `GET /api/files/{file_url}/markdown`

本轮不包含：

- `GET /api/download`
- `GET /api/rag/files/preview`
- `POST /api/files/{file_url}/markdown`

## 已完成实现

### 1. FastAPI 原生读服务层扩展

在 [`ai_actuarial/api/services/read.py`](/c:/Projects/AI_actuarial_inforsearch/ai_actuarial/api/services/read.py) 中新增：

- `list_sources(...)`
- `get_file_detail(...)`
- `get_file_markdown(...)`

实现原则：

- 直接复用 `Storage` 现有读取能力
- 不通过 HTTP 再转调 legacy Flask
- 对外返回结构保持与当前 Flask 路由一致

### 2. FastAPI 原生读路由扩展

在 [`ai_actuarial/api/routers/read.py`](/c:/Projects/AI_actuarial_inforsearch/ai_actuarial/api/routers/read.py) 中新增：

- `GET /api/sources`
- `GET /api/files/detail`
- `GET /api/files/{file_url}/markdown`

权限保持与 Flask 对齐：

- `/api/sources` -> `files.read`
- `/api/files/detail` -> `files.read`
- `/api/files/{file_url}/markdown` -> `markdown.read`

### 3. 兼容性处理

这轮特别处理了一个容易踩坑的点：

- `query` 参数里的 `url` 会保留原始 `%20`
- `path` 参数里的 `file_url` 在 FastAPI 中会被额外解码

因此 markdown 路由没有直接使用 FastAPI 传入的 `file_url`，而是从原始请求路径中恢复编码片段，再做一次受控解码。这样可以兼容：

- 文件 URL 本身包含空格
- 文件 URL 本身包含字面量 `%20`

这比当前 legacy Flask markdown 路由更稳，不会把所有 `%20` 都错误地还原成空格。
